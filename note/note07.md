> # ステージング

---

## > ## 基本定義

**ステージング**とは、
`git commit` するために、変更を一時的に「準備（stage）」しておく領域のこと。

この領域は **ステージングエリア（index）** とも呼ばれる。

---

## > ## 状態の3段階

Git では、1つのファイルは以下の3状態のどれかにある：

| 状態       | 説明                 |
| -------- | ------------------ |
| 作業ディレクトリ | ファイルを編集した状態        |
| ステージング   | コミットに含める変更を指定した状態  |
| コミット済み   | 履歴として Git に記録された状態 |

---

## > ## ステージングの目的

* **複数ファイルの変更から、必要なものだけを選んでコミット**できる
* **意味のある単位でコミット**できる（例：修正・機能追加・削除を分ける）
* **変更履歴をきれいに保つ**ための前処理

---

## > ## 基本コマンド

```bash
git add ファイル名
```

これにより、そのファイルの現在の状態がステージングされる。

---

## > ## よく使うパターン

| コマンド                         | 意味                     |
| ---------------------------- | ---------------------- |
| `git add ファイル名`              | ファイルの変更をステージングに追加      |
| `git add .`                  | カレントディレクトリ以下をすべて追加     |
| `git reset ファイル名`            | ステージから外す（ファイルの内容はそのまま） |
| `git restore --staged ファイル名` | 同上（新しいGitではこちらが推奨）     |

---

## > ## ステージング状態の確認

```bash
git status
```

出力例：

```text
Changes not staged for commit:
  modified:   example.py

Changes to be committed:
  new file:   config.yaml
```

---

## > ## コミットとの関係

```bash
git commit -m "メッセージ"
```

この時点で、**ステージングされた内容だけがコミット**される。
ステージされていない変更は無視される。

---

## > ## ステージングの可視化（VSCode）

1. 左側の「Source Control」アイコンをクリック（Ctrl + Shift + G）
2. ファイルを選んで \[+] ボタン → ステージングされる
3. ステージされた変更は「Staged Changes」に表示される
4. 差分を確認した上で、右上のチェックマークでコミット可能

---

## > ## 例：変更を一部だけコミットしたい

```bash
git add -p
```

対話的に変更を1ブロックずつ選んでステージング可能。

---

> ## 補足：ステージングなしに直接コミットはできる？

```bash
git commit -a
```

→ **ステージングをスキップして、変更されたすべてのトラッキング済みファイルをコミット**する。
※新規ファイルは含まれない。

---

## > ## まとめ

* ステージングは「コミットする内容を明示的に選ぶ」ための操作
* 差分の粒度を整え、履歴を綺麗に保つ
* VSCode・CLI どちらでもステージング操作は可能
